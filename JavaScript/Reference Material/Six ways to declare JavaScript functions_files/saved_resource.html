                      Šime Vidas • a year ago

                      If you don’t mind, I’d like to comment on the way this article is structured and presented 😅. It follows a pattern that is quite common in technical articles, where the content is categorized and comprehensive. This format is, I think, not well suited for situations where the reader needs to retrieve practical advice from the article.

                      Hear me out. What the reader needs is advice on which patterns to use in certain scenarios and use cases. So, “to do thing X, use pattern A”, “for use case Y, use pattern B”. If you remember Douglas Crockford’s JavaScript sessions at Yahoo, he was very good at giving this type of advice. He would focus on explaining the important language features and patterns.

                      In that light, “all the ways to declare a function” isn’t very useful. It requires the reader to process the article thoroughly, while taking notes, which they then have to store somewhere. If the reader at some later point needs practical advice on which function pattern to use, they have to look at their notes instead of this article (not optimal), and if they lose their notes, they have to come back to the article and re-read the entire text, in order to refresh their memory and find the solution somewhere in the text.

                      If the article was instead structured with an emphasis on use cases and usage examples, the reader could use it regularly to find the best solution for whatever situation they’re in. I hope, I’ve explained my point well 😋.

                          golden number Šime Vidas • a year ago

                          I use function declarations (function A(){};) when a function expression is not appropriate or when it is important that that a function is hoisted.

                          I use function declaration generators (function* B(){};) when I want to exit and then re-enter a function.

                          I use anonymous function expressions (var C = function(){};) when I want to pass a function as an argument to another function or I want to form a closure.

                          I use function expression generators (var D = function* [name](){};) when I want to exit and then re-enter a nested function.

                          I use arrow function expressions (() => {};) when I want to lexically bind the 'this' value.

                          I use named function expressions (var E = function F(){};) when I am doing recursion or want to see the function name in the debugger.

                          I used immediately-invoked function expressions (var F = (function(){return function(){}})();) when I want to use the module pattern.

                              Dmitri Pavlutin Mod Šime Vidas • a year ago

                              Hey Sime,
                              Your advice is very interesting. Thanks for your time writing this.

                              In my opinion, learning from: "for this case -> use this type" is not solid. It may be useful for specific fixed situations, but it cannot teach you to see the "global view". Trying to understand the "global view" is harder, but is a better investment in time. It will help you understand the problem and make decisions by yourself.
                              And the number of situations is very big and it is complicated to find a pattern to describe them all.

                              In other words, my articles approach with categories and "global view" is harder to take from the start, but is a good investment for a full understanding.
                              Your approach with “to do thing X, use pattern A” is easier to start with, but creates a "half of knowledge", which is not perfect for a long-time investment and solid language knowledge.

                              Thanks Sime :).
